# Lead Management Foundation

## Purpose

This backend foundation turns the existing Lead Management screens into an operational Supabase-backed workflow without creating a second Lead system.

`public.partner_patient_referrals` remains the original partner submission and attribution record.

`public.leads` becomes the internal AfraMedico operational Lead record.

One partner referral may produce a maximum of one Lead through:

`leads.source_referral_id = partner_patient_referrals.id`

## Tables

Created by:

`supabase/migrations/20260719100000_lead_management_foundation.sql`

Tables:

- `public.leads`
- `public.lead_notes`
- `public.lead_activities`
- `public.lead_documents`

Additive uniqueness constraints:

- `public.partners (id, organization_id)`
- `public.partner_patient_referrals (id, organization_id)`
- `public.leads (id, organization_id)`

## Lead State Model

Lead state is separated into three fields:

- `lead_status`: lifecycle state
- `pipeline_stage`: operational workflow stage
- `qualification_status`: review outcome

Initial `lead_status` values:

- `open`
- `on_hold`
- `converted`
- `closed`

Initial `pipeline_stage` values:

- `new_lead`
- `contact_attempted`
- `contacted`
- `qualification`
- `medical_records_pending`
- `clinical_review`
- `hospital_matching`
- `quotation_received`
- `patient_decision`
- `converted`
- `closed`

Initial `qualification_status` values:

- `unreviewed`
- `qualified`
- `not_qualified`
- `more_information_required`

## Codes

`lead_code` and `patient_reference_code` are generated by database-side functions.

The frontend must not generate authoritative Lead codes, patient reference codes, or unsafe sequence values.

Examples:

- `LEAD-2026-A1B2C3D4`
- `PAT-2026-B5C6D7E8`

## Organization Isolation

All Lead tables include `organization_id`.

RLS follows the existing ICOS convention:

`organization_id = public.current_organization_id()`

`public.current_organization_id()` reads:

`auth.jwt().app_metadata.organization_id`

Partner Portal users must not receive `organization_id` in JWT metadata and must not receive direct RLS access to:

- `leads`
- `lead_notes`
- `lead_activities`
- `lead_documents`

Partner access remains mediated through trusted Edge Functions.

## Data Normalization

The backend intentionally avoids compatibility JSON columns for:

- activity
- attachments
- communications
- reminders
- patient cases

Mapping rules:

- Notes go to `lead_notes`
- Activity and simple communication logs go to `lead_activities`
- File metadata goes to `lead_documents`
- Current reminder need maps to `next_follow_up_at`
- Case conversion maps to `converted_case_id`
- Coordinator, hospital, and referral partner display assignments are stored in transitional display-name fields while future selectors are connected to normalized IDs.

The frontend mapping layer temporarily reconstructs the legacy `Lead` UI shape from normalized database rows.

## Contact Validation

At least one meaningful contact method is required:

- `primary_email`
- `phone_e164`
- `whatsapp_e164`

Empty strings are normalized to `null` where practical so the check constraint is reliable.

Email format validation remains in the application/service layer.

## Development Data

`src/data/leads.json` and browser localStorage are development data only.

They are not imported during normal production startup.

If historical demo Leads must be loaded into a development database, use a deliberate one-time import script or manual SQL process. The import process must be idempotent and should use `lead_code` or another stable external identifier to prevent duplicates.

## Current Frontend Integration

The existing Lead screens are preserved:

- Lead Dashboard
- Lead Directory
- Lead Pipeline
- Lead Profile
- Add New Lead

The frontend now uses `src/services/leadService.ts` as the mapping boundary.

Live Supabase mode:

- reads from `public.leads`
- creates Leads in `public.leads`
- persists Lead activity to `public.lead_activities`
- persists notes to `public.lead_notes`

Development fallback:

- allowed only when Supabase environment variables are missing locally
- disabled in production builds
- does not silently combine live Supabase rows with localStorage rows
